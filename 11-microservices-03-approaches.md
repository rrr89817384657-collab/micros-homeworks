# Домашнее задание к занятию «Микросервисы: подходы»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.


## Задача 1: Обеспечить разработку

Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка. 
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- облачная система;
- система контроля версий Git;
- репозиторий на каждый сервис;
- запуск сборки по событию из системы контроля версий;
- запуск сборки по кнопке с указанием параметров;
- возможность привязать настройки к каждой сборке;
- возможность создания шаблонов для различных конфигураций сборок;
- возможность безопасного хранения секретных данных (пароли, ключи доступа);
- несколько конфигураций для сборки из одного репозитория;
- кастомные шаги при сборке;
- собственные докер-образы для сборки проектов;
- возможность развернуть агентов сборки на собственных серверах;
- возможность параллельного запуска нескольких сборок;
- возможность параллельного запуска тестов.

Обоснуйте свой выбор.

### Решение: GitLab, собственные GitLab Runner, Docker.

- Облачная система используем GitLab. Это готовая облачная платформа, которую не нужно разворачивать и администрировать самостоятельно. Все компоненты работают в облаке, доступны через браузер и API.

- Система контроля версий в GitLab поддерживает из коробки. 

- Репозиторий на каждый сервис. Под каждый микросервис можно заводить отдельный проект с собственным репозиторием. Можно группировать их по продуктовым направлениям внутри групп.

- Запуск сборки по событию из системы контроля версий - настраивается в файле .gitlab-ci.yml. Можно указать правила: запускать сборку при пуше в определенные ветки, при создании merge request или при появлении тега.

- Запуск сборки по кнопке с указанием параметров. В GitLab можно передать любые параметры (переменные окружения), которые повлияют на выполнение сборки.

- Возможность привязать настройки к каждой сборке - настройки CI/CD хранятся в файле .gitlab-ci.yml в корне репозитория. Этот файл версионируется вместе с кодом, поэтому каждый коммит имеет именно те настройки, которые были актуальны на момент его создания. Сборка всегда выполняется по правилам, соответствующим версии кода.

- Возможность создания шаблонов для различных конфигураций сборок - в GitLab есть механизмы include и extends. Можно вынести общие части конфигурации в отдельные YAML-файлы и подключать их в разных проектах. 

- Возможность безопасного хранения секретных данных - в настройках проекта или группы есть раздел Variables. Туда можно добавить переменные (пароли, токены, ключи доступа) с двумя важными опциями: Masked (значение скрыто в логах) и Protected (переменная доступна только для защищенных веток, например main. Можно подключить HashiCorp Vault.

- Несколько конфигураций для сборки из одного репозитория - в одном .gitlab-ci.yml можно описать несколько разных сценариев.

- Кастомные шаги при сборке - в секции script можно писать любые команды, которые выполнятся в контейнере: установка пакетов, запуск скриптов, компиляция, прогон тестов, сборка Docker-образов, отправка уведомлений. Нет никаких ограничений.

- Собственные докер-образы для сборки проектов - GitLab включает встроенный Container Registry. Мы можем собирать свои образы и хранить их там же. В конфигурации сборки указываем image: registry.gitlab.com/название-проекта/образ:версия, и раннер будет использовать именно этот образ.

- Возможность развернуть агентов сборки на собственных серверах - устанавливаем GitLab Runner на свои серверы. Регистрируем их в проекте на GitLab.com, указываем тип исполнителя Docker. Раннеры будут забирать задания с облачного GitLab и выполнять их на нашем железе. Это снимает ограничения по времени сборки и позволяет использовать собственные мощности.

- Возможность параллельного запуска нескольких сборок - настраивается через конфигурацию раннеров. Можно зарегистрировать несколько раннеров, и GitLab будет распределять задачи между ними. Также можно у одного раннера в настройках указать concurrent для количества одновременно выполняемых задач.

- Возможность параллельного запуска тестов - в .gitlab-ci.yml есть ключевое слово parallel для разбивки выполнения параллельных джоб. GitLab сам распределит нагрузку, а результаты потом соберет в один отчет. Также можно использовать матричные сборки для тестирования на разных версиях или конфигурациях.

#### Обоснование выбора

Я выбрал GitLab, потому что это  решение, которое закрывает все требования в рамках одного продукта. Мне не нужно связывать GitHub с Jenkins через вебхуки, настраивать права доступа отдельно для кода и отдельно для CI/CD, синхронизировать пользователей. В GitLab доступ к CI/CD автоматически наследуется от доступа к репозиторию.
Удобная конфигурация через YAML-файл в репозитории. Все настройки рядом с кодом, под версионным контролем. Если что-то сломалось в сборке, я могу откатиться на предыдущий коммит и получить ровно ту конфигурацию, которая работала раньше.
Docker обеспечивает чистоту экспериментов. Каждая сборка стартует в свежем контейнере, нет накопленного мусора, конфликтов зависимостей, проблем с версиями. А встроенный Container Registry позволяет хранить кастомные образы тут же. Параллельные тесты и сборки ускоряют разработку.
